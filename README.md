# ETLQA_E2E_Framework

## Overview

`ETLQA_E2E_Framework` is a **metadata‚Äëdriven end‚Äëto‚Äëend ETL testing framework** built with Python.  It validates data movement from source systems (Oracle DB or CSV files) into a Microsoft SQL Server target.  The framework leverages:

- **Pandas** for in‚Äëmemory data manipulation
- **SQLAlchemy** (with `cx_Oracle` and `pyodbc`) for database connectivity
- **PyTest** for automated test execution
- **Logging** (file + console) for traceability
- **HTML reports** generated by `pytest‚Äëhtml`

The design is intentionally lightweight so it can be dropped into any HRM‚Äëstyle application where you need to ensure that data extracted from Oracle or CSV files lands correctly in a SQL Server data‚Äëwarehouse.

## Getting Started

### Prerequisites

1. **Python 3.10+**
2. Access to the source systems:
   - Oracle DB DSN (`ORACLE_DSN` environment variable) ‚Äì e.g. `user/pass@host:1521/service`
   - CSV files under `data/csv/`
3. Access to the target Microsoft SQL Server (`SQLSERVER_DSN` environment variable) ‚Äì a typical ODBC connection string.
4. (Optional) **Virtual environment** ‚Äì recommended to isolate dependencies.

### Installation

```bash
# Clone the repository (if you haven't already)
git clone https://github.com/Veera26-git/ETLQA_E2E_Framework.git
cd ETLQA_E2E_Framework

# Create a venv and install dependencies
python -m venv venv
source venv/bin/activate   # on Windows: venv\Scripts\activate
pip install -r requirements.txt
```

### Environment Variables

Create a `.env` file at the project root (or export the variables in your shell):

```dotenv
ORACLE_DSN=user/password@host:1521/ORCL
SQLSERVER_DSN=Driver={ODBC Driver 17 for SQL Server};Server=server;Database=db;UID=user;PWD=pass
CSV_ROOT=data/csv
```

The framework will automatically pick them up via `src/config.py`.

## Running the Framework

### CLI Execution

```bash
python -m src.etl.main
```

The script iterates over every table defined in `metadata.yaml`, extracts source data, pulls the target data, runs all configured validations and logs a detailed summary.  It exits with status `0` when **all** validations succeed, otherwise `1`.

### Test Suite

The repository ships with a comprehensive PyTest suite that validates the validation logic itself.  It mocks the target loader to avoid requiring a live SQL Server.

```bash
pytest                # Generates an HTML report at reports/report.html
```

The HTML report is produced by the `pytest‚Äëhtml` plugin (configured in `pytest.ini`).

## Metadata‚ÄëDriven Configuration

`metadata.yaml` is the single source of truth for what tables are under test and which validations should run.  Example snippet:

```yaml
tables:
  employees:
    source:
      type: csv
      path: employees.csv
    target:
      type: sqlserver
      table: dbo.employees
    validations:
      - count
      - null
      - duplicate
      - schema
    schema:
      id: int
      name: str
      hire_date: datetime64[ns]
      salary: float
```

- **source.type** ‚Äì `csv` or `oracle`
- **target.type** ‚Äì currently only `sqlserver`
- **validations** ‚Äì list any of `count`, `aggregate`, `data`, `duplicate`, `null`, `schema`
- **schema** ‚Äì column name ‚Üí expected pandas dtype (used by the schema validator)

Add new tables or validation types by editing this file ‚Äì the framework picks up changes automatically.

## Extending the Framework

1. **New Source Types** ‚Äì add a new extractor in `src/etl/extractor.py` and expose it via `extract_source`.
2. **New Target Types** ‚Äì add a loader in `src/etl/loader.py` and extend the dispatch in `load_target`.
3. **Additional Validations** ‚Äì implement a function in `validation.py`, register it in the `VALIDATORS` map, and list its name in `metadata.yaml`.
4. **Custom Reports** ‚Äì the HTML report is generated by pytest; you can add further pytest plugins (e.g., `pytest‚Äëcov` for coverage) via `requirements.txt`.
